The Problem: USCIS Forms Are PDFs From Another Era

- USCIS releases PDFs in XFA format. 
- Mention user pain points: Can't fill it on browser, users have to use Acrobat
- Even on Acrobat, some fields aren't usable
- How to sign - can't use Digital Signatures
(Show Reddit Posts)


Steps

1. Convert the PDF from XFA to AcroForm

- XFA is not supported by browsers.
- AcroForms can be read + filled by pdf-lib.
- AcroForms give me stable field names.

The Process:
- Download the official USCIS PDF.
- Open in Adobe Acrobat Pro.
- Convert to AcroForm

 

2. Adding Custom Fields inside PDF 

Instead of relying on Acrobat’s auto-detection (which is wrong 40% of the time), I manually the input fields
(Show screenshot inside Acrobat)

And for each field, I assign a stable internal field name like:

applicant_last_name
physical_address_city
travel_planned_yes_no
passport_expiration_date

We use these in our schema. More on that later.

By controlling the field names myself, I create:

  • a universal naming convention
  • a stable mapping layer
  • schema portability across Fillvisa Free and Fillvisa Plus

 

This is the most time-consuming step.

 

3. Rebuilding the Entire Form As a Web Replica

The PDF UI is unusable. So I recreate every page in a clean, Bootstrap-style layout:

  • Labels
  • Inputs
  • Radio groups
  • Conditional fields
  • Repeating sections (like “List your previous employers”)

Each field mirrors the exact structure of the official PDF.

Conditional Logic Examples

“Have you used any other names?”
→ Reveal alias fields.

“Has anyone filed a petition for you?”
→ Reveal details section.

“Is your mailing address same as physical address?”
→ Hide or show mailing form.

Validation

I implement validation similar to USCIS expectations:

Required fields

Date formatting

Character limits

Conditional required fields

Phone and email formats

Everything is client-side + instant.

 

4. Autosave That Works Everywhere

Every time the user types anything, Fillvisa checks whether the data changed:

function saveFormToLocalStorage() {
    const current = getFormData();
    if (JSON.stringify(current) !== JSON.stringify(lastSaved)) {
        localStorage.setItem("formData-3", JSON.stringify(current));
        lastSaved = current;
    }
}

Key features:

  • Debounced for performance
  • Saves only when something changes
  • Saves page-wise (formData-1, formData-2, …)

 

This means zero data loss - even if:

  • browser crashes
  • page reloads
  • mobile users switch tabs
  • laptop battery dies

 

5: Loading the Data Back on Page Load

When the user opens the form again:

function loadFormFromLocalStorage() {
    const saved = localStorage.getItem("formData-1");
    if (!saved) return;
    const data = JSON.parse(saved);
    Object.keys(data).forEach(id => {
        //fill the respective input
    });
}

The form reconstructs itself seamlessly.

This is what makes Fillvisa feel “magical” - you never lose progress.

 

 

6: Full-Form Validation Before PDF Output

Before users generate the final PDF:

  1. Fillvisa checks every page valid status (should be true):
    1. Every required field must be valid.
    2. Every conditional must be satisfied.
  2. Only then does the user reach the output page.

This avoids broken PDFs.

 

 

7: Mapping Local Variables Back to PDF Fields

This is where the schema becomes powerful.

I create a mapping file like:

{
  "applicant_last_name": "LastName_1",
  "applicant_first_name": "FirstName_1",
  "travel_planned_yes_no": "TravelYN",
  "passport_expiration_date": "PassExpDate"
}

Then in pdf-lib:

pdfForm.getTextField("LastName_1").setText(data.applicant_last_name);

 

The mapping layer ensures:

  • Every web field goes into the correct PDF field
  • Data types sync correctly
  • Radios/checks convert to USCIS field values (“Y”, “N”, “X”)
  • Format constraints (e.g., MM/DD/YYYY) are preserved

Once mapping is done → user downloads the official USCIS PDF, with their data filled in.

No backend. No server. No data uploaded.

How I Stay Updated When USCIS Updates Forms

One of the invisible challenges of building a USCIS form-filling tool is form updates.

USCIS regularly revises:

  • field labels
  • form structure
  • number of pages
  • version numbers
  • mandatory questions

If you miss an update, the tool becomes instantly outdated - and users suffer.

 

How I monitor changes

USCIS provides a form update notification system, and I subscribe to every update. Whenever USCIS releases:

  • a new edition date,
  • a new PDF version,
  • a new instructions PDF,
  • or retires an old edition,

I receive an immediate alert.

 

How updates flow into Fillvisa

When USCIS publishes a new edition:

  1. I download the new PDF.
  2. Compare the field structure with the current schema.
  3. Review the instructions PDF for new questions or rule changes.
  4. Regenerate the AcroForm field names if needed.
  5. Update the mapping layer.
  6. Rebuild the corresponding web page if fields changed.
  7. Publish the new version on Fillvisa.

This ensures Fillvisa remains aligned with the official, current USCIS standards.

 

What’s Next for Fillvisa

This post covers what Fillvisa is today. But there’s a lot more coming that will make the system more powerful, more consistent, and easier to extend.

Building a Universal Form Schema

Right now, each form has three partially separate layers:

  1. PDF schema -  the field names I create inside Acrobat
  2. Web form schema -  the IDs, validations, conditionals on the web interface
  3. PDF-mapping schema - the JSON mapping that connects web fields → PDF fields

It works well, but long-term, it should be one unified schema.

Why a universal schema matters

A single schema will allow:

  • Consistent field definitions across forms
  • Automatic validation generation
  • Automatic PDF mapping generation
  • Reusable conditional logic
  • Easier maintenance when USCIS updates forms
  • Faster building of new forms
  • Cleaner developer documentation (eventual API)
  • More reliable import (for Fillvisa Plus)

A unified schema will look like:

{
  "id": "applicant_last_name",
  "label": "Last Name",
  "type": "text",
  "required": true,
  "pdfField": "LastName_1",
  "validation": { "maxlength": 40 },
  "conditional": null
}

One definition → used everywhere.

Fillvisa will eventually have a schema directory containing JSON definitions for every field across all forms.
